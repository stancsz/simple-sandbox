# SOP: Automated Demand Prediction & Forecasting Workflow

This Standard Operating Procedure (SOP) outlines the lifecycle of the automated demand prediction system, integrated under the 'Ghost Mode' operations of Phase 29.

## Overview
The goal of the forecasting workflow is to autonomously track, analyze, and predict resource demand (e.g., token usage, API calls) and provide actionable recommendations aligned with the Corporate Strategy.

## Step 1: Metric Recording
1. **Trigger:** The system routinely records historical metrics via the `record_metric` tool from the `forecasting` MCP server.
2. **Frequency:** Metrics should be recorded at least daily, storing values like `llm_token_usage` and `api_calls` into the local `sqlite` database.
3. **Context:** Ensure the appropriate `company` context (e.g., `@simple-cli/showcase`) is used for multi-tenant isolation.

## Step 2: Demand Prediction
1. **Trigger:** The `forecast_demand` tool in the `business_ops` MCP server is called by the `Scheduler` (Ghost Mode) or manually by an analyst.
2. **Input Parameters:**
   - `metric_name`: e.g., 'llm_token_usage'
   - `forecast_horizon`: e.g., '30d'
   - `company`: e.g., '@simple-cli/showcase'
3. **Processing:**
   - The tool queries the forecasting DB for historical values.
   - Calculates trends ('increasing', 'decreasing', 'stable') using `simple-statistics`.
   - Generates a baseline forecast (`forecast_metric`).
   - Reads the latest `CorporateStrategy` from the `Brain` MCP memory.
   - Feeds the combined data into the LLM to generate recommendations and policy implications.

## Step 3: Caching & Recommendations
1. **Output Structure:** The tool outputs a JSON object containing:
   - `forecast_values` (with confidence bounds)
   - `trend`
   - `recommendations` (e.g., "Scale swarm capacity by 20%")
   - `policy_implications` (e.g., "Decrease max_agents_per_swarm from 10 to 5")
2. **Idempotency:** The forecast is cached in the `Brain`'s `EpisodicMemory` under a deterministic key for 24 hours to prevent redundant API calls and model executions.

## Step 4: Policy Updates (Ghost Mode)
1. The recommendations generated by `forecast_demand` can be directly routed to the `PolicyEngine` (or `proposeStrategicPivot` tool) for autonomous self-improvement.
2. If the policy implications indicate a runway exhaustion or high risk, they trigger a `convene_board_meeting` for strategic review.
