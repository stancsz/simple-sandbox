# Agency Spawning Protocol

The Agency Spawning Protocol enables a primary parent agency to autonomously spawn child agencies, inherit contexts, and allocate isolated resources.

## Workflow

1. **Context Initialization**: The parent agency extracts relevant niche-specific information (e.g., frontend_dev, data_science) from its `CorporateStrategy` or general context.
2. **Resource Allocation Check**: The parent verifies its resource limits and allocates a portion of its budget and processing resources to the new child agency. It checks for `yoloMode` and `resourceLimit` constraints.
3. **Environment Isolation**: A separate agent directory and `EpisodicMemory` (Brain) directory are generated for the child.
4. **Context Injection**: The context generated in Step 1 is injected into the child's workspace before starting.
5. **Federation Registration**: Once active, the child agency registers itself in the Distributed Ledger and the Federation Protocol. The parent agency can then discover its capabilities and delegate tasks.

## Merging and Retiring

As part of the Autonomous Ecosystem Evolution (Phase 36), the root parent agency can dynamically restructure the ecosystem based on utilization metrics and meta-learning insights.
* **Merging**: If multiple child agencies with the same role are underutilized, the Agency Orchestrator can merge them via the `merge_child_agencies` tool. This archives the source agency's environment and consolidates operations into the target agency.
* **Retiring**: If a child agency is failing frequently or has critically low token efficiency, it can be shut down via the `retire_child_agency` tool. This safely archives the child agency's memory and frees up resources for the ecosystem.

## Validation & Constraints

During validation testing (`tests/integration/phase32_agency_spawning_validation.test.ts`), the following configurations and constraints were identified:

1. **Resource Constraints & Policy Execution**: Child agencies must be spawned with a `resourceLimit` of at least `10`. Additionally, the parent agency consults its `CorporatePolicy` stored in `EpisodicMemory` to ensure the requested `resourceLimit` does not exceed the allowed `token_budget`. Attempting to spawn with fewer than 10 tokens or more than the budget will result in an `Insufficient resources or policy constraint violation` error.
2. **Network Reliability**: Synchronization with the Federation Protocol and Distributed Ledger during spawn is critical. If the `registerAgency` tool call fails due to a network error, it must be caught and gracefully handled by the parent agency without corrupting state.
3. **Unique Identifiers**: A child agency is identified by an internal ID (e.g., `agency_child1`). Attempting to spawn another agency with the same ID, or into an existing directory, will result in a `Child agency ID already exists` error.
4. **Isolated Memory & Context Injection**: Child agencies operate entirely inside their own designated `brain` directory for `EpisodicMemory`. Parent memory searches will not surface child records and vice versa. Upon spawn, the initial strategic context generated by the LLM is automatically injected as a `CorporateStrategy` record into the child's fresh memory store.
5. **Auditing**: Every successful spawn operation is permanently logged as an `autonomous_decision` back in the parent agency's memory to maintain an audit trail.

6. **Task Delegation Failure & Recovery**: During federated projects involving spawned agencies, network latency or timeouts can occur. The parent orchestrator correctly identifies the failure and can gracefully retry task delegation (`delegateTask`), successfully recovering without corrupting the overall federated project state.
7. **Federated Virtual Ledger**: Spawned agencies can reliably perform work and have their contributions accurately tracked in the multi-agency `distributed_ledger`. Stress tests validate that parent agencies properly record net-negative values as service consumers, while child agencies accumulate correct net-positive values.
8. **Cross-Agency Pattern Recognition**: The root parent agency can periodically scan the collective experiences of its spawned child agencies using the Brain MCP's `cross_agency_pattern_recognition` tool. This meta-orchestration capability analyzes patterns across all child agency namespaces, uses the LLM to synthesize common successes or recurring failures, and stores the resulting `cross_agency_pattern` back into memory. This continuous collective learning shapes future strategic pivots and improves the success rates of new child agencies.
9. **Meta-Learning Parameter Adjustment**: Following Cross-Agency Pattern Recognition, the resulting ecosystem insights can be applied via the `apply_ecosystem_insights` tool in the `agency_orchestrator`. This tool dynamically reads new `ecosystem_policy` proposals from the Brain, translates them via LLM into exact configuration parameters (like `scaling_threshold` and `max_agents`), and applies those directly into the `swarm_config:<agency_id>` entries of the active child agencies. This forms a complete recursive optimization loop for multi-agency performance tuning.

## Ecosystem Optimization (Meta-Learning)

Child agencies spawned via this protocol benefit from root-level meta-learning optimizations. The `apply_ecosystem_insights` tool in the Agency Orchestrator dynamically retrieves the latest ecosystem-wide insights (`ecosystem_policy`) from the Brain's Episodic Memory. It then updates the `swarm_config:<agency_id>` for the relevant child agencies. Subsequent operations, such as SOP execution, automatically query and inject these optimized parameters (e.g., `max_agents`, `routing_latency_ms`) to ensure peak operational efficiency across the swarm fleet.
